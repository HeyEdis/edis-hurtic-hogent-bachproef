# Ralph Init

Scaffold the Ralph autonomous e2e test generation loop into the current repository. This creates the scripts and prompt templates needed to run Codex in an iterative loop that autonomously writes Playwright e2e tests from pre-generated user stories and plan.json files.

Reference: https://www.aihero.dev/tips-for-ai-coding-with-ralph-wiggum

## Pipeline

This skill is designed to work as the final step in this pipeline:

1. **generate-user-stories** → reads `RouteConfig.tsx`, generates user story markdown files in `apps/client/playwright/user-stories/`
2. **user-story-to-plan-json** → reads a user story, generates `apps/client/playwright/plans/<story>.json`
3. **ralph-init** (this skill) → scaffolds the loop scripts to execute the plans

**Prerequisites:** Steps 1 and 2 must be completed before running Ralph. Ralph does NOT generate user stories or plan.json files — the user is expected to have generated those beforehand.

### Key artifacts

| Artifact | Location | Generated by |
|---|---|---|
| User stories | `apps/client/playwright/user-stories/<role>/<story>.md` | `generate-user-stories` skill |
| Plan files | `apps/client/playwright/plans/<story>.json` | `user-story-to-plan-json` skill |
| Progress logs | `apps/client/playwright/plans/<story>.progress.md` | Ralph (at runtime) |
| Spec files | `apps/client/playwright/e2e/<story>.spec.ts` | Ralph (at runtime) |
| POM pages | `apps/client/playwright/support/pages/<feature>Page.ts` | Ralph (at runtime) |
| Fixtures | `apps/client/playwright/fixtures/<feature>.fixtures.ts` | Ralph (at runtime) |

## Steps

1. Check what already exists — don't overwrite any existing Ralph files without asking
2. Verify prerequisites:
   - `apps/client/playwright/user-stories/` must have user story files
   - `apps/client/playwright/plans/` must have at least one plan.json file
   - If either is missing, tell the user to run the prerequisite skills first
3. Scan `apps/client/playwright/plans/` to count total plans and plans with incomplete tasks
4. Create the following files (skip any that already exist and are valid):

### `scripts/ralph.sh`

An executable bash script that:

- Takes an iteration count as the first argument (default: 5)
- **Accepts `--story=<name>` to work on a specific plan** (optional):
  - `--story=jobs-detail` → plan at `apps/client/playwright/plans/jobs-detail.json`, user story resolved from the plan's `source.user_story` field, progress at `apps/client/playwright/plans/jobs-detail.progress.md`
- **Without `--story`**: scans `apps/client/playwright/plans/` for all `.json` files with incomplete tasks (`"passes": false`), processes them sequentially in alphabetical order
- Loops N times, each time calling `codex exec` with the iteration prompt, passing the plan file, user story, and progress log as context
- After each iteration, checks stdout for `<promise>COMPLETE</promise>` — if found (all tasks in current plan complete), moves to the next incomplete plan or exits if all plans are done
- Includes a `--hitl` flag that runs one iteration, then writes the prompt context to a temp file and starts `codex` in interactive TUI mode so the user can review and give feedback within the same session
- Supports `--afk` flag for fully autonomous mode — implies `--yolo`, uses the AFK-specific prompt (`prompts/ralph-iteration-afk.md`) that instructs Ralph to make all decisions independently
- Supports `--sandbox` flag to use Codex's built-in sandbox (`--sandbox danger-full-access`) for controlled execution
- Supports `--yolo` flag to run with `--yolo` (`--dangerously-bypass-approvals-and-sandbox`) for fully autonomous operation (no approval prompts, no sandbox)

Example structure:

```bash
#!/usr/bin/env bash
set -e

PLANS_DIR="apps/client/playwright/plans"
ITERATIONS=5
STORY=""
HITL=false
AFK=false
SANDBOX=false
YOLO=false

while [[ $# -gt 0 ]]; do
  case "$1" in
    --hitl) HITL=true; ITERATIONS=1 ;;
    --afk) AFK=true; YOLO=true ;;
    --sandbox) SANDBOX=true ;;
    --yolo) YOLO=true ;;
    --story=*) STORY="${1#*=}" ;;
    [0-9]*) ITERATIONS="$1" ;;
  esac
  shift
done

CODEX_FLAGS="--sandbox workspace-write"
if [ "$YOLO" = true ]; then
  CODEX_FLAGS="--yolo"
fi

PROMPT_FILE="prompts/ralph-iteration.md"
if [ "$HITL" = true ]; then
  PROMPT_FILE="prompts/ralph-iteration-hitl.md"
elif [ "$AFK" = true ]; then
  PROMPT_FILE="prompts/ralph-iteration-afk.md"
fi

# Build list of plan files to process
if [ -n "$STORY" ]; then
  PLAN_FILES=("${PLANS_DIR}/${STORY}.json")
  if [ ! -f "${PLAN_FILES[0]}" ]; then
    echo "Error: plan not found: ${PLAN_FILES[0]}" >&2
    exit 1
  fi
else
  # Find all plans with incomplete tasks (has "passes": false), sorted alphabetically
  PLAN_FILES=()
  for f in $(ls "${PLANS_DIR}"/*.json 2>/dev/null | sort); do
    if grep -q '"passes": false' "$f"; then
      PLAN_FILES+=("$f")
    fi
  done
  if [ ${#PLAN_FILES[@]} -eq 0 ]; then
    echo "All plans complete! Nothing to do."
    exit 0
  fi
  echo "Found ${#PLAN_FILES[@]} plan(s) with incomplete tasks."
fi

# Process each plan
for PLAN in "${PLAN_FILES[@]}"; do
  STORY_NAME=$(basename "$PLAN" .json)
  PROGRESS="${PLANS_DIR}/${STORY_NAME}.progress.md"

  # Resolve user story path from plan's source.user_story field
  USER_STORY=$(grep -o '"user_story": "[^"]*"' "$PLAN" | head -1 | cut -d'"' -f4)

  # Create progress file if it doesn't exist
  if [ ! -f "$PROGRESS" ]; then
    echo "# Ralph Progress Log — ${STORY_NAME}" > "$PROGRESS"
    echo "" >> "$PROGRESS"
    echo "Each iteration appends what was done, decisions made, and files changed." >> "$PROGRESS"
  fi

  echo "=== Processing plan: ${STORY_NAME} ==="

  for ((i=1; i<=ITERATIONS; i++)); do
    # Check if this plan still has incomplete tasks
    if ! grep -q '"passes": false' "$PLAN"; then
      echo "Plan ${STORY_NAME} complete!"
      break
    fi

    echo "--- Iteration $i/$ITERATIONS for ${STORY_NAME} ---"

    PROMPT=$(cat "$PROMPT_FILE")
    CONTEXT="@${PLAN} @${PROGRESS}"
    if [ -n "$USER_STORY" ] && [ -f "$USER_STORY" ]; then
      CONTEXT="@${USER_STORY} ${CONTEXT}"
    fi

    if [ "$HITL" = true ]; then
      # Run the task non-interactively first
      result=$(codex exec $CODEX_FLAGS "$CONTEXT $PROMPT" | tee /dev/stderr)
      # Then start interactive session for review
      echo "\n=== Task complete. Starting interactive review session ==="
      echo "Paste feedback or type 'done' to continue to next task."
      codex $CODEX_FLAGS
    elif [ "$SANDBOX" = true ]; then
      result=$(codex exec --sandbox danger-full-access "$CONTEXT $PROMPT" | tee /dev/stderr)
    else
      result=$(codex exec $CODEX_FLAGS "$CONTEXT $PROMPT" | tee /dev/stderr)
    fi

    if [ "$HITL" != true ] && [[ "$result" == *"<promise>COMPLETE</promise>"* ]]; then
      echo "Plan ${STORY_NAME} complete!"
      break
    fi
  done
done

# Final check: are ALL plans now complete?
ALL_DONE=true
for f in $(ls "${PLANS_DIR}"/*.json 2>/dev/null); do
  if grep -q '"passes": false' "$f"; then
    ALL_DONE=false
    break
  fi
done

if [ "$ALL_DONE" = true ]; then
  echo "All plans complete!"
fi
```

### `prompts/ralph-iteration.md`

The prompt for a single Ralph iteration. It should instruct Codex to:

1. Read the e2e testing skill file (`apps/client/playwright/skill/e2e-testing-guidelines.md`) — follow ALL conventions
2. Read the plan file and progress log to understand what's done and what remains
3. Read the referenced user story for context on the feature being tested
4. Pick the highest-priority incomplete task (`"passes": false`), using this priority order:
   - Page render and navigation tests (prove the page loads)
   - Core interaction tests (click, fill, select)
   - State and validation tests (enabled/disabled, checked/unchecked)
   - Edge case tests
5. Explore the codebase to understand the relevant React components before writing tests:
   - Read the page component referenced in the user story
   - Follow child components to identify interactive elements
   - Check which `data-testid` attributes already exist
   - Check existing POM pages, fixtures, and routes for reusable infrastructure
6. Implement the first (highest-priority) task as the tracer bullet:
   - Add `data-testid` attributes to React source components if missing
   - Create or update the POM page in `apps/client/playwright/support/pages/`
   - Create or update the fixture in `apps/client/playwright/fixtures/`
   - Add missing routes to `apps/client/playwright/support/data/routes.ts`
   - Write the spec in `apps/client/playwright/e2e/`
   - Run the spec file: `cd apps/client && npx playwright test e2e/<spec-file>.spec.ts`
   - Fix failures and re-run until it passes — do NOT continue until green
   - Update the progress log, mark `"passes": true`, commit
7. Implement each remaining task in priority order:
   - Same steps as above (without the spec run after each)
   - Update the progress log, mark `"passes": true`, commit each task separately
8. Once ALL tasks are implemented, run the full spec file: `cd apps/client && npx playwright test e2e/<spec-file>.spec.ts`
9. Fix any failures and re-run until the full spec file is green
10. When the spec file passes, run the full test suite: `cd apps/client && npm run test:e2e`
11. If the full suite passes, output `<promise>COMPLETE</promise>`
    - If the full suite fails, do NOT output `<promise>COMPLETE</promise>` — fix the failures first

The prompt should be concrete and specific, not vague. IMPORTANT: The prompt must instruct Ralph to read the e2e skill files at runtime for conventions.

Include this quality guidance in the prompt:

```markdown
QUALITY:
- One commit per task — keep changes small and focused
- If a task feels too large, break it into subtasks and complete them in sequence
- Do not take shortcuts — follow the patterns already established in this codebase
- Leave the codebase better than you found it
- When modifying existing test files, reuse the existing test setup — never duplicate POM pages or fixtures
- Read the e2e-testing-guidelines skill file carefully before writing any test code

VERIFICATION:
- After implementing ALL tasks, run the spec file: `cd apps/client && npx playwright test e2e/<spec-file>.spec.ts`
- Fix any failures and re-run until the full spec file is green
- Only after the spec file passes, run the full suite: `cd apps/client && npm run test:e2e`

E2E CONVENTIONS (read the full skill file for details):
- Use `data-testid` locators ONLY — never locate by visible text (this is a multilingual app with lingui/i18n)
- Use `page.getByTestId()` exclusively
- Banned locators: getByText, getByRole with name, getByLabel, getByPlaceholder, CSS selectors, XPath
- Sole exception: page.getByRole('dialog') or page.getByRole('alert') for third-party components
- Assert on visibility, enabled state, checked state, and input values — NEVER assert on text content
- Safe assertions: toBeVisible(), not.toBeVisible(), toBeEnabled(), toBeChecked(), toHaveValue(), toHaveURL()
- Unsafe assertions (NEVER use): toHaveText(), toContainText()
- Routes come from apps/client/playwright/support/data/routes.ts — add missing routes, never hardcode
- Test data persisted to the database must be prefixed with E2E_
- Use date-fns for date manipulation — never raw Date arithmetic
- Tests hit the real staging API — no mocks except for third-party external services
- Each test must be independently runnable — no shared state, no execution-order dependency
```

### `prompts/ralph-iteration-hitl.md`

The HITL prompt (used when `--hitl` flag is set). This is identical to `ralph-iteration.md` (including all quality and e2e convention sections) with one addition at the end — a **HITL Review Phase** section:

```markdown
## HITL Review Phase

After completing and committing the task above, **do not exit**. Simply tell the user: "Task complete. Ready for your review." Then wait.

### Step 1: Code review loop

When the user provides feedback about the test:
1. Fix the test code based on their feedback (wrong assertions, bad locator strategy, POM structure, data-testid naming).
2. Re-run the specific spec file: `cd apps/client && npx playwright test e2e/<spec-file>.spec.ts`
3. Commit the fix.
4. Tell the user: "Fixed and committed. Anything else?"
5. Repeat until the user approves.

### Step 2: Introspect

Once the code is approved, run the introspect skill (`apps/client/playwright/skill/introspect.md`).
This reviews the session for friction — especially any review remarks — and applies targeted
improvements.

Key rules for e2e introspect scope:
- E2e skill files (`apps/client/playwright/skill/*.md`) — update if a convention was missing or wrong
- Shared test data (`routes.ts`, `constants.ts`, `sharedTypes.ts`) — update if new shared data was discovered
- POM pages and helpers — refactor if patterns emerged that improve reusability
- Do NOT update files outside `apps/client/playwright/` except for adding `data-testid` to React components
- Prompt sync — keep all three prompts/ralph-iteration*.md variants consistent

If the user had no remarks and the session was smooth, skip this step.
```

IMPORTANT: The review phase must NOT run git commands or generate a summary. The user reviews the code themselves. Codex just waits for feedback.

### `prompts/ralph-iteration-afk.md`

The AFK prompt (used when `--afk` flag is set). This is a variant of the standard iteration prompt with one key difference: Ralph must make all decisions autonomously and never block on human input. It includes everything from `ralph-iteration.md` (including the quality guidance and e2e conventions) plus these additional instructions:

```markdown
AUTONOMOUS MODE — You are running unattended. Never stop to ask for human input.

DECISION MAKING:
- When you encounter a test design decision (POM structure, fixture composition,
  locator strategy, test granularity, data-testid naming), make the best decision
  you can based on existing patterns in the codebase and the e2e-testing-guidelines skill file.
- If a `data-testid` attribute is missing from a React component, add it following
  the naming convention: `<feature>-<element>` (static) or `<feature>-<element>-<id>` (dynamic).
- If a route is missing from `routes.ts`, add it.
- If a decision is blocked by missing information that cannot be inferred from
  the codebase or user story, document the blocker in the progress log, skip that task,
  and move on to the next one.

RISK MANAGEMENT:
- Prefer simple, readable tests over clever abstractions.
- When uncertain between locator strategies, follow the e2e-testing-guidelines strictly.
- When adding `data-testid` to React components, place it on the actual rendered DOM element,
  not a wrapper that gets stripped.

INTROSPECT:
After completing and committing the task, run the introspect skill
(`apps/client/playwright/skill/introspect.md`). Apply all improvements autonomously:
1. Friction log — review what went wrong (retried commands, failed locators, missing data-testid).
2. Skill audit — check if e2e skill files need updates.
3. Shared data updates — update routes.ts, constants.ts, sharedTypes.ts if new shared data was discovered.
4. Prompt sync — keep all three prompts/ralph-iteration*.md variants consistent.
Commit introspect changes separately (message: "chore: introspect — <brief summary>").
```markdown
5. Make `scripts/ralph.sh` executable
6. Print a ready-to-run summary tailored to what was detected. Build the **exact command** the user should run based on:
   - The number of plan files found in `playwright/plans/`
   - The total number of incomplete tasks across all plans
   - Suggest iterations = total incomplete tasks + a few buffer

   The summary should look like this:

   ```markdown
   Ralph e2e loop is ready!

   📁 Plans directory: apps/client/playwright/plans/
   📋 Incomplete plans: X of Y
   📝 Total incomplete tasks: Z

   Recommended first run (HITL — watch and refine one story):

     ./scripts/ralph.sh --story=signin --hitl --yolo

   Process all incomplete plans autonomously:

     ./scripts/ralph.sh <Z+5> --afk

   Process a specific story autonomously:

     ./scripts/ralph.sh 10 --story=jobs-detail --afk

   --afk mode will:
   • Skip all permission prompts (implies --yolo)
   • Make test design decisions autonomously (POM structure, locator strategy, data-testid naming)
   • Add data-testid attributes to React components as needed
   • Run introspect after each task

   After returning from AFK, review progress logs: apps/client/playwright/plans/*.progress.md

   Use --sandbox for untrusted or overnight runs.
   Delete progress logs when a story's tests are stable — they're session-specific.

   Want me to kick off the first HITL iteration now?
   ```

   Adapt the commands to the actual plan count and task count. Always suggest starting with `--hitl` first. Always ask whether to start the first run — don't start it automatically.
